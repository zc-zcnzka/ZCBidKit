# ZC投标工具箱开发说明

## 目标

本文面向参与 `client/` 独立客户端开发的成员。阅读后应能快速判断：代码放在哪里、Renderer 和 Electron Main 如何协作、哪些组件和样式应复用、改动后如何验证，以及需要查看哪些仓库级关联目录。

## 技术栈与命令

- Electron：桌面窗口、本地文件系统、配置、后台任务、导出和更新检查。
- Vite + React + TypeScript：Renderer 层 UI 与交互编排。
- Radix UI：Dialog、Tooltip、Switch 等无样式基础组件。
- CSS：全局 CSS，不使用 Tailwind，不复用旧 `frontend/` 样式。

常用命令：

```powershell
cd client
npm install
npm run dev
npm run build
npm audit
```

项目包含 Electron Main 侧 native 依赖，例如 `better-sqlite3`。`npm install` / `npm ci` 后会通过 `postinstall` 执行 `electron-builder install-app-deps`，把 native 模块重建到当前 Electron 版本。若出现 `NODE_MODULE_VERSION` 不匹配或 IPC 大面积未注册，先运行：

```powershell
cd client
npm run postinstall
```

## 架构边界

- `client/` 是独立桌面客户端，不依赖旧 `frontend/` 和 `backend/` 源码路径。
- 本项目是安装在客户本地计算机上的一体化客户端，Renderer 和 Electron Main 不是传统前后端网络边界；两侧传递的业务参数默认视为客户端内部可信数据，不要因为 IPC 往返而做多余、重复的校验或重复计算。
- Renderer 不直接访问 Node、`fs`、`path`、`ipcRenderer`，只通过 `window.yibiao` 调用 preload 暴露的安全 API。
- Electron Main 负责本地能力：配置、文件解析、AI 请求、SQLite Store、后台任务、导出、自动更新。
- IPC 文件只注册通道和转发参数；业务逻辑放 `electron/services/*.cjs`。
- Renderer 页面只做 UI 和交互编排，不承载耗时任务生命周期，也不保存业务权威状态。
- Prompt builder 只负责组装消息，不直接请求 AI；不要在组件内硬编码大段 Prompt。
- 所有用户可见文案使用中文。

## 目录职责

```text
client/
├── electron/                 # Electron Main / Preload / IPC / Node 服务
│   ├── ipc/                  # IPC 通道注册，只做参数转发
│   ├── services/             # 配置、AI、文件、Store、任务、导出等业务服务
│   └── utils/                # Main 侧工具函数
├── src/
│   ├── app/                  # 路由、菜单、全局 Provider
│   ├── components/           # 应用布局组件，例如 AppShell、Sidebar
│   ├── features/             # 业务功能模块
│   ├── shared/               # 跨功能共享能力
│   ├── main.tsx              # Renderer 入口
│   └── styles.css            # 全局样式
└── assets/                   # 客户端图标和静态资源
```

关联目录入口：

- `analytics/`：独立 Cloudflare Workers 埋点服务和统计看板。客户端只通过共享埋点入口上报；修改埋点字段、页面映射、公告读取或统计展示时才需要查看。
- `sql/workspace_schema.sql`：`workspace/yibiao.sqlite` 目标完整表结构说明。运行时不会执行该文件；实际建表和升级以 `client/electron/services/sqliteDatabase.cjs` 的 migration 为准。

新增主菜单页面时，通常同步修改：

- `src/shared/types/navigation.ts`
- `src/app/menuConfig.ts`
- `src/app/AppRouter.tsx`
- 如需页面级工具条，再按页面状态渲染 `FloatingToolbar`

如一级菜单需要承载多个子功能，在 `menuConfig.ts` 为该一级 `AppMenuItem` 配置 `children`，点击一级菜单后由 `shared/ui/SecondaryMenuPage.tsx` 在右侧主内容区渲染二级菜单页；不要在侧边栏做展开式或悬浮式二级菜单。二级入口仍需补充 `SectionId`、具体页面路由和埋点看板中文映射。

## Feature 与 Shared

业务功能放在 `src/features/<feature-name>/`：

```text
features/<feature-name>/
├── pages/        # 页面级组件
├── components/   # 本功能专用组件
├── hooks/        # 本功能专用 Hook
├── services/     # Renderer 侧轻量编排或兼容工具；长任务不要放这里
└── types.ts      # 本功能专用类型
```

当前功能模块：

- `technical-plan`：技术方案。
- `knowledge-base`：知识库。
- `duplicate-check`：标书查重。
- `rejection-check`：废标项检查。
- `settings`：设置页，管理文本模型、生图模型、文件解析和关于信息。

共享能力放在 `src/shared/`：

- `shared/ai/aiClient.ts`：Renderer 侧 AI 请求门面，仅用于仍在 Renderer 侧的轻量请求。
- `shared/prompts/`：仍由 Renderer 使用的 Prompt builder。已迁入 Main 后台服务的流程不要保留 Renderer 直连 AI 占位 Prompt。
- `shared/types/`：跨功能类型，包含 `ipc.ts` 中的 `window.yibiao` bridge 类型。
- `shared/ui/`：跨功能 UI，例如 `FloatingToolbar`、`SecondaryMenuPage`、`DetailHelpLink`、`MarkdownRenderer`、Toast。
- `shared/utils/`：跨功能纯工具，例如 ID、字数统计。

共享层规则：

- `shared/` 不引用任何 feature 模块，避免循环依赖。
- 类型优先使用 `interface`；联合类型和泛型工具类型使用 `type`。
- 新增 bridge API 时，同步 `electron/ipc/*`、`electron/preload.cjs` 和 `src/shared/types/ipc.ts`。

## Electron Main、IPC 与 Store

- `electron/` 使用 CommonJS，保持 `.cjs`。
- `electron/ipc/*.cjs` 只注册/转发 IPC，不写业务逻辑。
- `electron/services/*.cjs` 承载 Main 侧业务逻辑。
- Renderer bridge 以 `src/shared/types/ipc.ts` 为类型权威；不要在多个文件里各自手写不一致的返回类型。
- Main 侧文件读写显式使用 UTF-8，并把 Windows 中文路径作为默认场景处理。

核心服务分工：

- `configStore.cjs`：配置读写，保存到 Electron `userData/user_config.json`。
- `aiService.cjs`：文本模型、生图模型、JSON 修复和模型请求统计。
- `fileService.cjs`：文档导入与解析，统一按配置选择本地或 MinerU 解析方式。
- `sqliteDatabase.cjs`：工作区 SQLite 初始化和 runtime migration。
- `technicalPlanStore.cjs`、`duplicateCheckStore.cjs`、`rejectionCheckStore.cjs`、`knowledgeBaseStore.cjs`：各功能 SQLite Store。
- `taskService.cjs`：通用后台长任务管理、任务组锁、active task、事件推送和状态落盘。
- `exportService.cjs`：Word 导出、Markdown 转 Word、图片和 Mermaid 导出。

新增文件上传、导入、转换功能时，优先复用 `fileService.cjs` 的 `parseDocumentWithConfig(app, filePath, config, { assetScope })`。删除业务资源时，同步调用 `electron/utils/importedImages.cjs` 的 `deleteImportedImageBatches(app, assetScopePrefix)` 清理导入图片资产。

## 数据存储

- 配置保存到 Electron `userData/user_config.json`。
- 业务工作区保存到 Electron `userData/workspace/`。
- 结构化业务状态统一进入 `userData/workspace/yibiao.sqlite` 或功能专用 Store。
- SQLite 表结构调整时，同步更新 runtime migration 和仓库根目录 `sql/workspace_schema.sql`，后者只用于阅读、评审和排查问题。
- 大文本 Markdown、原始上传文件和图片资产保存为文件，SQLite 只保存路径、hash、计数和结构化状态。
- Renderer 只允许使用 `localStorage` 保存轻量 UI 偏好；大文本、草稿、API Key、流程状态都走 Main 侧 Store/IPC。
- 技术方案正文展示和 Word 导出以 `outlineData.outline[*].content` 为权威来源。
- 知识库旧数据迁移由进入知识库页面后触发，使用项目内 Radix Dialog 提示，不用系统弹窗；只迁移旧版 `status = success` 的文档，未完成或处理中的旧文档不会迁移到新版本知识库。具体迁移方案见 `client/doc/sqlite改造方案_知识库.md`。

技术方案流程状态约定：

- 主流程 Step 只表示页面级阶段，例如 `document-analysis`、`bid-analysis`、`outline-generation`、`global-facts`、`content-edit`。
- Step 内部如果存在可中断子流程，不新增主 Step；应在 Store 中落盘为该 Step 的子状态，页面加载后按子状态恢复 UI。
- Step01 多标段待选择属于 `document-analysis` 的 pending 子状态，临时 Markdown 保存为 `tender-pending-*.tmp.md`，SQLite 只保存恢复弹窗需要的文件名、解析器、标段列表和临时文件相对路径，Renderer 不接触临时文件路径。
- Step05 正文生成属于 `content-edit` 的后台任务子流程，运行态、章节状态、规划结果和 runtime JSON 由 Main 持续写入 Store，用于异常关闭后的暂停恢复。
- 新增 Step 内子状态时，要同时定义异常关闭后的恢复、确认完成后的清空、取消后的清理和重置工作区时的清理规则。

## 后台任务

- 所有耗时 AI、文件解析、批处理、生成、检查、导出转换都必须放在 Electron Main 后台任务中执行。
- Renderer 只负责启动任务、订阅事件、读取 Store 快照和展示状态。
- Main 任务启动后必须立即写入 running 状态，并在每个关键阶段持续写入对应 Store。
- 页面卸载不能取消任务，也不能作为任务失败信号。
- 页面重新挂载后必须先读取 Store，再订阅任务事件，并通过 `getActiveTasks()` 触发 active task 状态回放。
- 如果 Store 中存在 running/pausing 等运行态，但 Main 没有对应 active task，应标记为“上次任务未完成，请重新执行”或进入业务约定的可恢复状态，不能静默恢复为 idle。

任务组约定：

| 任务组 | 锁策略 | 说明 |
| --- | --- | --- |
| `technical-plan` | `group-exclusive` | Step02/Step03/Step04 共用技术方案工作区，组内互斥。 |
| `duplicate-check` | `group-exclusive` | 标书查重外层是一轮分析任务，内部子流程可以并发。 |
| `rejection-check` | `group-exclusive` | 废标项解析和检查共用同一检查缓存，组内互斥。 |
| `knowledge-base` | `scope-exclusive` | 后续接入统一任务服务时按 `documentId` 加锁；当前仍由 `knowledgeBaseService.cjs` 独立管理文档级互斥。 |

清空规则原则：

- 重新生成某一步时，只清空该步骤旧结果和后续步骤结果，不清空前置输入。
- 清空动作应在 Main 侧任务确认启动后执行并落盘，Renderer 不要在 IPC 成功前清空持久化业务数据。
- 目录重新生成、编辑、添加或删除后，必须清空旧正文内容和正文生成缓存。

## UI 与组件复用

- UI 使用全局 CSS + Radix 基础组件，不使用 Tailwind。
- 新页面优先复用现有卡片、胶囊 Tab、状态栏、工作区、列表/树 + 阅读器、`FloatingToolbar` 等模式，不另起一套视觉语言。
- 页面专用样式只补充必要结构和间距，不重写整套颜色、圆角、阴影和字号体系。
- 页面根容器保持 `height: 100%` / `min-height: 0`；长内容在页面内部滚动，禁止依赖 `body` 全局滚动。
- `FloatingToolbar` 是覆盖层，不要为它额外预留大段空白；必要内容通过内部滚动查看。
- 用户确认、长文案提示、危险操作优先使用项目内 Radix Dialog 和统一按钮样式；新增或重构时不要使用 `alert`，也尽量不要使用系统 `window.confirm` / `window.prompt`。
- 成功、失败、警告、普通消息统一使用 `shared/ui/ToastProvider`，不要自建全局提示。

Markdown 约定：

- Markdown 展示统一使用 `shared/ui/MarkdownRenderer.tsx`。
- Markdown 编辑统一使用 `shared/ui/MarkdownEditor.tsx`。
- 展示远程公告、AI 输出或其他不可信内容时，必须传 `allowRawHtml={false}`。
- 只有明确展示本地解析出的可信 HTML 文档结构时，才允许开启 raw HTML。
- Mermaid 图以 Markdown `mermaid` 代码块保存；Renderer 本地渲染预览，Word 导出由 Main 转图片。

## AI、Prompt 与日志

- Prompt 放 `src/shared/prompts/` 或 Main 侧对应服务中，避免在组件内硬编码大段 Prompt。
- 已迁入 Electron Main 后台服务的流程，不保留 Renderer 直连 AI 占位 service 或 Prompt。
- JSON 主请求解析失败时复用现有 JSON 修复链路；修复请求只提交模型已返回内容片段、校验问题和目标格式，不重复提交完整大文本上下文。
- 后台任务不得按 token、chunk 或部分内容落盘；只在阶段开始、单次 AI 请求完整返回、失败、暂停或任务结束时写入 Store。
- Renderer 不展示 token 级增量结果；页面只展示任务阶段、进度、日志和完整结果。
- 开发者模式下 AI 请求日志保存到 Electron `userData/logs/ai/`；不要把日志路径或敏感内容暴露给普通用户。
- 开发者模式下的程序调试日志统一使用 `electron/utils/developerLog.cjs` 写入 `userData/logs/<module>/*.jsonl`；关闭开发者模式时必须是 no-op，日志写入失败不能影响主流程。
- 程序调试日志只保存在本机，不自动上传服务器；默认记录阶段、计数、耗时、hash、warning 和错误信息，不主动写 API Key、Token、Base URL、Prompt、AI 响应、正文全文或完整输出路径。
- 排查文件解析、查重、废标检查、导出等模块问题时，优先使用模块级 JSONL 日志，不把详细调试过程塞进 SQLite 任务日志。

## 埋点统计

- 关联实现位于仓库根目录 `analytics/`：`analytics/worker/` 是上报与查询 API Worker，`analytics/dashboard/` 是统计看板。部署和密钥说明见 `analytics/README.md`。
- Renderer 埋点入口统一使用 `src/shared/analytics/analytics.ts`。
- AI 模型请求统计由 Main 侧 `electron/services/aiService.cjs` 统一上报。
- 禁止以清理、重构、简化为由删除或绕过现有埋点、统计字段、页面访问映射、模型使用统计和看板展示。
- 新增主菜单页面时，同步更新 `SectionId`、`menuConfig.ts`、`AppRouter.tsx`，并补 `analytics/dashboard/public/src/pages/traffic.js` 页面中文名映射。
- 埋点路由和值应稳定、可读、低基数，不包含用户输入、文件名、本地路径、文档 ID、任务 ID、随机 ID 或时间戳。
- 禁止上传 `api_key`、`base_url`、`mineru_token`、Prompt、AI 响应、错误详情、文件名、本地路径、招标文件内容、正文草稿等敏感或大文本数据。
- 埋点失败必须静默处理，不影响用户主流程。

## 发布与打包

- 打包配置位于 `client/package.json` 的 `build` 字段。
- Release workflow 位于 `.github/workflows/release.yml`，推送 `v*` tag 或手动输入 tag 触发。
- workflow 在 `client/` 下 `npm ci`，从 tag 同步 `package.json` 版本，再用 `electron-builder --publish never` 构建并通过 `gh release upload` 上传产物。
- 当前未接入代码签名；Windows/macOS 未签名提示是已知发布约束，不要在普通功能改动里临时绕过。

本地打包命令：

```powershell
cd client
npm run dist:win
npm run dist:mac
```

## 验证标准

常规 Renderer / TypeScript 改动：

```powershell
cd client
npm run build
```

Electron Main / preload 改动：

```powershell
cd client
node --check electron\preload.cjs
node --check electron\services\<file>.cjs
npm run build
```

涉及窗口、IPC、文件选择、导出、后台任务事件时，还需要手动运行：

```powershell
cd client
npm run dev
```

涉及依赖变更时额外运行：

```powershell
cd client
npm audit
```

涉及 SQLite native 执行路径时，普通 Node 可能因 Electron ABI 不匹配无法跑运行时 smoke；`node --check` 只做语法检查，真实 Store 冒烟应使用 Electron 运行时。

## 开发约定

- 优先小步修改，不做无关重构。
- 保持 Windows 中文路径和中文编码兼容。
- 不新增全局状态库，除非实际复杂度需要。
- 不在 Renderer 存储 API Key。
- 不为新上传功能另写一套文档解析链路。
- 命名保持清晰：页面 `*Page.tsx`，服务 `*Service.ts` / `*.cjs`，Hook 以 `use` 开头。
- 用户可见文案、Toast、Dialog 标题和按钮文案保持中文、清晰、可操作。
- 新增主菜单入口时，必须按“埋点统计”章节完成埋点和看板映射自查。
